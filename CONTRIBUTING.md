# Contributing to MAOS

MAOS is a living specification. It grows through critique, not through applause.

## What is most valuable

**Critique.** Open an issue titled `Challenge: [topic]` if you think something in the spec is wrong, impractical, or missing. The best contributions to MAOS so far came from adversarial analysis — finding the gaps, not praising the walls.

**Adversarial review.** Try to break the security model. Pick a mechanism, assume you are an attacker with full knowledge of the spec (Kerckhoffs' principle), and report what you find. See the [Adversarial Assurance Review](companion/MAOS_Adversarial_Assurance_Review_v5_8.md) for prior findings.

**Implementation experience.** If you implement any part of MAOS — even just the Commit Layer or Capability Tokens — report what worked and what didn't. Implementation feedback is worth more than spec text.

**Spec improvements.** Submit a pull request with your proposed change and a short rationale. Changes that simplify the spec are especially welcome. The spec has grown through addition; it needs people who can subtract.

**Formal verification.** TLA+, Alloy, or other model-checking contributions for the Mandatory Rules and Security Invariants would significantly strengthen the architecture.

## How to submit

- **Issues** for questions, critique, and adversarial findings
- **Pull requests** for spec changes (one change per PR, with rationale)
- **Discussions** for open-ended architectural questions

## Language

Contributions are welcome in English or German.

## What this project is not

MAOS is not a commercial product, not a startup, and not seeking investment. It is one person's attempt to think through a problem that the industry hasn't solved yet. Contributions are welcomed in that spirit.

## Code of conduct

Be constructive. Be honest. Be specific. Disagree with the architecture, not with the person.
